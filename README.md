# Hugo Infragistics

Bilingual (EN/JA) marketing site built with [Hugo](https://gohugo.io) and the [Hugolify](https://www.hugolify.io) framework. Japanese content is automatically generated from English sources via a local Ollama translation pipeline orchestrated by n8n.

## Tech Stack

- **Static site generator:** Hugo Extended v0.146.4
- **Theme:** [Hugolify](https://www.hugolify.io) v1.27.14 (via Hugo modules)
- **CSS:** Bootstrap v5 + SASS, compiled with PostCSS (Autoprefixer, CSSNano, PurgeCSS)
- **Deployment:** Netlify (auto-deploy on push to `main`)
- **Translation:** n8n workflow → local Ollama → GitHub API

## Prerequisites

- Hugo Extended ≥ 0.146.4
- Go ≥ 1.24
- Node.js ≥ 20
- Yarn

## Local Development

```bash
# Install dependencies
yarn install

# Start dev server
yarn watch

# Production build
yarn build
```

## Project Structure

```
content/en/        # English content (source of truth)
content/ja/        # Japanese content (auto-generated — do not edit manually)
data/              # YAML data files (menus, banner, footer nav)
assets/sass/       # SASS stylesheets
layouts/           # Custom Hugo template overrides
static/            # Static assets (images, etc.)
config/_default/   # Hugo configuration
```

## Content

Content lives under `content/en/` as Markdown files with YAML frontmatter.
The homepage (`_index.md`) uses a block-based layout system defined entirely
in frontmatter — no markdown body.

```
content/en/
├── _index.md               # Homepage (block-based)
├── blog/
│   ├── _index.md           # Blog listing
│   └── *.md                # Blog posts
├── contact/_index.md
├── legal-mentions/_index.md
├── privacy-policy/_index.md
└── sitemap/_index.md
```

**Do not edit `content/ja/` files directly.** They are overwritten by the
translation workflow. Edit `content/en/` and re-run the workflow.

## Translation Workflow (n8n + Ollama)

Japanese content is generated by an n8n workflow running on a LAN machine.
See [`N8N_TRANSLATION_PLAN.md`](N8N_TRANSLATION_PLAN.md) for the full
architecture, node structure, prompt templates, and field-by-field translation
rules.

**Summary:**
1. Manually trigger the n8n workflow
2. Workflow compares `content/en/` GitHub blob SHAs against `translated_from_sha`
   in existing `content/ja/` files
3. New or changed files are translated via Ollama (`qwen2.5:14b` recommended)
4. Translated files are committed back to `content/ja/` via GitHub API
5. Push triggers automatic Netlify build and deploy

## Deployment

Netlify deploys automatically on every push to `main`.
Configuration is in [`netlify.toml`](netlify.toml).

| Context | Command |
|---|---|
| Production | `yarn build` (minified, git info enabled) |
| Deploy preview | `hugo --gc --minify --buildFuture` |
| Branch deploy | `hugo --gc --minify` |

## Hugo Modules

Dependencies are managed as Hugo modules in `go.mod`:

| Module | Purpose |
|---|---|
| hugolify-theme v1.27.14 | Core layouts and components |
| hugolify-admin v0.9.2 | Headless CMS interface |
| hugo-mod-bootstrap-scss | Bootstrap v5 |
| Splide, Tobii, Rellax | UI components |

Update modules:
```bash
yarn update      # Hugo modules only
yarn update:all  # Full dependency update
```
